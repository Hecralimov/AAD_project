### A18 – @OneToOne Mapping: User ↔ CustomerProfile

`CustomerProfile.java` (lines 22–25):
```java
@OneToOne(fetch = FetchType.LAZY)
@JoinColumn(name = "user_id", referencedColumnName = "id", unique = true)
@JsonIgnore
private User user;
```

✅ The `@OneToOne` mapping is correctly declared with a `UNIQUE` FK constraint on `user_id`. `FetchType.LAZY` is appropriate. `@JsonIgnore` prevents circular serialization.

**⚠️ However:** The `User` entity does **not** declare the inverse side of this relationship (`@OneToOne(mappedBy = "user")`). The association is unidirectional. This is not broken, but means you cannot navigate from `User` → `CustomerProfile` via JPA — only via `CustomerProfileRepository.findByUserEmail()`. Acceptable for now but worth noting.

**Verdict: ✅ PASS** (unidirectional is functional)

## Category 3 — Security Implementation (A5, A14, A15)

### A5 – GlobalExceptionHandler

`GlobalExceptionHandler.java` is annotated `@RestControllerAdvice` (line 13) and handles:

| Handler | HTTP Code | Line |
|---|---|---|
| `RuntimeException` | 400 Bad Request | 17–28 |
| `IllegalArgumentException` | 400 Bad Request | 31–42 |
| `AccessDeniedException` | **403 Forbidden** | 45–56 |

**⚠️ FAIL – Missing 404 Handler:** There is no handler for `NoSuchElementException`, `EntityNotFoundException`, or a custom `ResourceNotFoundException`. Any "not found" case currently falls through to the `RuntimeException` handler, which returns **400 Bad Request** instead of **404 Not Found**. This is semantically incorrect and violates A5's requirement for graceful 404 handling.

**Verdict: ⚠️ PARTIAL PASS / FAIL**
- ✅ 403 handled — `GlobalExceptionHandler.java:45`  
- ✅ 500 handled — `GlobalExceptionHandler.java:59`  
- ❌ **404 not handled** — `GlobalExceptionHandler.java` has no `@ExceptionHandler` for any "not found" exception type

### A15 – JWT Required for All Paths Except /api/auth/**

`SecurityConfig.java` (lines 51–63):
```java
.authorizeHttpRequests(auth -> auth
    .requestMatchers("/api/auth/**").permitAll()
    .requestMatchers("/api/individual/**").hasRole("INDIVIDUAL")
    .requestMatchers("/api/admin/**").hasRole("ADMIN")
    .requestMatchers("/api/corporate/**").hasRole("CORPORATE")
    .requestMatchers("/api/products/**").permitAll()         // ⚠️ See below
    .requestMatchers(HttpMethod.GET, "/api/categories/**").permitAll()
    .requestMatchers(HttpMethod.GET, "/api/orders/**").hasRole("INDIVIDUAL")
    .requestMatchers(HttpMethod.POST, "/api/orders/**").hasRole("INDIVIDUAL")
    .requestMatchers("/v3/api-docs/**", "/swagger-ui/**", "/swagger-ui.html").permitAll()
    .requestMatchers(HttpMethod.GET, "/api/analytics/corporate/**").hasRole("CORPORATE")
    .requestMatchers(HttpMethod.GET, "/api/analytics/individual/**").hasRole("INDIVIDUAL")
    .anyRequest().authenticated())
```

**⚠️ Issue 1 – `/api/products/**` is fully `permitAll()`**: This means `POST /api/products`, `PUT /api/products/{id}`, and `DELETE /api/products/{id}` in `ProductCatalogController` are **publicly accessible** with no authentication. Any anonymous user can create, modify, or delete products. This is a **security hole**.
- File: `SecurityConfig.java:56`, `ProductCatalogController.java:34–48`

**⚠️ Issue 2 – `GET /api/orders/**` restricted to INDIVIDUAL but `GET /api/orders` and `GET /api/orders/{id}` in `OrderController` are generic admin-style endpoints**: These broad list endpoints should arguably be ADMIN-only, but are restricted to INDIVIDUAL. Not a security flaw per se, but an inconsistency.

**Verdict: ⚠️ PARTIAL PASS / FAIL**
- ✅ `/api/auth/**` is `permitAll()`  
- ✅ Role-based restrictions exist for admin/corporate/individual paths  
- ✅ `anyRequest().authenticated()` is the fallback  
- ❌ **`/api/products/**` (POST/PUT/DELETE) is unprotected** — `SecurityConfig.java:56`

## Category 4 — Role-Based Access Control (A13, A16)

### ADMIN – Global Analytics Access

`AnalyticsController.java` is mapped to `/api/admin/analytics/**` (line 24).  
`AdminAnalyticsController.java` is also mapped to `/api/admin/analytics` (line 14).

`SecurityConfig.java:54`: `.requestMatchers("/api/admin/**").hasRole("ADMIN")` — all `/api/admin/**` paths require ADMIN.

**Verdict: ✅ PASS** — ADMIN restriction is enforced at the `SecurityConfig` level.

**⚠️ However:** Neither `AnalyticsController` nor `AdminAnalyticsController` use `@PreAuthorize` annotations at the method level. Authorization is **URL-pattern-only**. This is functional but fragile — if a controller is ever mapped to a new path, the security won't follow it automatically. Method-level security (`@EnableMethodSecurity` + `@PreAuthorize("hasRole('ADMIN')")`) is the more robust pattern.

---

### CORPORATE – Store-Specific Inventory Management

`CorporateOperationsController.java` is mapped to `/api/corporate/operations/**` (line 27).  
`SecurityConfig.java:55`: `.requestMatchers("/api/corporate/**").hasRole("CORPORATE")` — correctly restricts to CORPORATE role.

`CorporateOperationsController.getInventory()` (line 43–48): Calls `corporateService.getStoreInventory(user.getId(), ...)`, which scopes products to the authenticated user's store.

`CorporateAnalyticsController.java` is mapped to `/api/analytics/corporate/**` (line 19).  
`SecurityConfig.java:61`: `.requestMatchers(HttpMethod.GET, "/api/analytics/corporate/**").hasRole("CORPORATE")` — correctly restricts.

**⚠️ FAIL – `CorporateOperationsController` uses `CorporateAnalyticsService` for operations**: The controller imports and uses `CorporateAnalyticsService` (lines 21, 32) for inventory and order management methods (`getStoreInventory`, `getStoreOrders`, `updateOrderStatus`). These are operational methods mixed into an analytics service. This is an architectural layering violation — inventory management should be in a dedicated `CorporateOperationsService` or `ProductService`/`OrderService`. This was noted as the root cause in a prior conversation (conversation `48d10250`).

**Verdict: ⚠️ PARTIAL PASS / FAIL**
- ✅ CORPORATE role restriction is enforced — `SecurityConfig.java:55`  
- ✅ Store scoping is enforced in the service layer — `CorporateAnalyticsService.java:60–63, 86–88`  
- ❌ **Inventory/order operations live in `CorporateAnalyticsService`** — architectural violation — `CorporateOperationsController.java:32`, `CorporateAnalyticsService.java:65–92`
